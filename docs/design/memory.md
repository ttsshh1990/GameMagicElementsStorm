`docs/design/current-design.md` 只保存最新设计共识；历史过程保存在这里。

## 2026-04-23 - 先建立设计工作方式

- 来源：项目初始化讨论。
- 摘要：用户希望先完善指导 AI agent 的文档，而不是直接进入游戏细节。
- 影响：设计区保持极简，只保留 `current-design.md` 和 `memory.md`。
- 取舍：不建立复杂的设计文档拆分区，避免早期维护成本过高。

## 2026-04-23 - 设立 Design Agent

- 来源：项目初始化讨论。
- 摘要：当任务是游戏设计脑暴时，应启用专门的 design agent，与用户讨论并维护设计文档。
- 影响：design agent 负责把每次有价值的讨论总结提炼到本文件。
- 取舍：当前设计只保留在 `current-design.md`，迭代过程保留在本文件，避免混淆最新共识和历史想法。

- 来源：用户提供的历史草稿 `/Users/tiantian/Downloads/元素弹幕_Roguelike_设计记录_v2.md`。
- 摘要：将草稿中已经完成的内容作为项目第一版正式设计基线导入，核心方向确定为“元素弹幕射击 + 局内 Roguelike 构筑”，build 乐趣围绕元素收集、元素合成、元素强化和符文激活展开。
- 影响：`docs/design/current-design.md` 从空白模板更新为第一版正式设计，当前已确定三基础元素、三阶段合成成长、元素强化方向、符文激活逻辑，以及 10 个组合的当前完成状态。
- 取舍：这次导入把草稿视为当前共识基线，而不是仅供参考的发散材料；但草稿中明确标记为待定、待重做、待补充的内容仍保留为未决问题，没有被强行补完。

## 2026-04-24 - 调整三元素组合记录方式与雷系技能归属

- 来源：用户在当前 Codex App thread 中对三元素组合表的反馈。
- 摘要：用户不希望用“战斗定位”约束技能，因为理论上每个技能都可以发展成主要核心。三元素组合表改为强调技能形态、视觉效果和造成伤害方式。
- 影响：`docs/design/current-design.md` 删除三元素组合表中的“定位”列；雷雷雷从电球改为闪电链；原雷雷雷的电球技能转移给雷雷冰。
- 取舍：后续设计不再优先按“主输出 / 辅助 / 控制定位”分类技能，而是先确定技能看起来是什么、如何命中、如何造成伤害，再通过元素强化和符文让任意技能具备核心 build 潜力。

## 2026-04-24 - 确定连续生存结构与统一数值规则

- 来源：当前 Codex App thread 中关于可玩初代原型所需细节的讨论。
- 摘要：取消波次制，改为怪物无限出现并逐渐变强；玩家在 2D 俯视角大平面房间中移动走位，技能以内置 CD 自动释放；怪物掉落经验晶体，玩家拾取后升级，升级时暂停弹出奖励；3 级后解锁元素合成。
- 影响：`docs/design/current-design.md` 增加战斗结构、经验升级、奖励节奏和数值规则。数值系统以 `BASE_DPS`、`TARGET_TTK`、`ENEMY_HP_BASE` 等基准参数驱动，技能、怪物、经验和成长主要使用公式与倍率计算。
- 取舍：原型优先验证连续生存压力、拾取经验和升级 build 决策，不采用每波结束给奖励的节奏；数值避免为每个技能每一级 hardcode，后续调参优先调整全局基准值和成长倍率。

## 2026-04-25 - 确定基础元素攻击方式

- 来源：当前 Codex App thread 中用户对火 / 冰 / 雷基础攻击的定义。
- 摘要：火元素基础攻击为小火球，攻击单个敌人；冰元素基础攻击为寒冰新星，以一个点为圆心爆开小范围寒霜，造成群伤和轻度减速；雷元素基础攻击为小闪电，攻击一个敌人后弹射 2-3 个目标，并附带短暂麻痹。
- 影响：`docs/design/current-design.md` 增加“基础元素攻击”表，明确 1-2 级未合成阶段的基础输出方式。
- 取舍：三个基础元素的基础 DPS 应基本一致，差异主要来自视觉形态、命中方式和轻量控制；冰的群伤与减速、雷的弹射与麻痹都不应导致基础输出明显高于火。
